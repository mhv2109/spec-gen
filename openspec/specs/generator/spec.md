## Requirements

### Requirement: OpenSpecValidatorValidation

The system SHALL validate OpenSpecValidator according to these rules:
- Spec documents must contain a title (# heading)
- Spec documents should contain a Purpose section
- Spec documents should contain either Requirements or Domains section
- Generated specs should not contain delta markers ([ADDED], [MODIFIED], [REMOVED])
- Markdown structure must be valid with no skipped heading levels
- Requirements must use RFC 2119 keywords (SHALL, MUST, SHOULD, MAY)
- Requirements should start with 'The system' or 'The [component]'
- Requirements should be at least 5 words long
- Scenarios must have exactly 4 hashtags (####) in the heading
- Scenarios must contain GIVEN, WHEN, and THEN clauses
- AND clauses in scenarios should be bolded (**AND**)
- GIVEN must come before WHEN, and WHEN must come before THEN
- Config must be an object
- schema must be a string if present
- context must be a string if present
- rules must be an object if present
- rules keys must be one of proposal, specs, design, tasks
- rules values must be arrays
- spec-gen.version should be a string if present
- spec-gen.generatedAt should be a string if present
- spec-gen.domains should be an array if present
- spec-gen.confidence should be a number if present

#### Scenario: ValidateSpecDocumentStructure
- **GIVEN** An OpenSpec document content
- **WHEN** validateSpecStructure() is called
- **THEN** Returns validation results with errors for missing title and delta markers, and warnings for missing
sections and malformed links

#### Scenario: ValidateRequirementFormat
- **GIVEN** A requirement string
- **WHEN** validateRequirementFormat() is called
- **THEN** Returns validation results with warnings for missing RFC 2119 keywords, incorrect imperative form,
and brief requirements

#### Scenario: ValidateScenarioFormat
- **GIVEN** A scenario string in Given/When/Then format
- **WHEN** validateScenarioFormat() is called
- **THEN** Returns validation results with errors for missing clauses, incorrect heading format, and
out-of-order clauses, and warnings for improperly formatted AND clauses

#### Scenario: ValidateConfig.yamlStructure
- **GIVEN** A configuration object
- **WHEN** validateConfigYaml() is called
- **THEN** Returns validation results with errors for invalid types and structure, and warnings for unknown
keys and missing recommended fields
